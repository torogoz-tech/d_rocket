# Migration guide: legacy packages → `d_rocket`

This guide covers every public symbol that the five legacy
packages (`d_serializer`, `d_serializer_builder`, `d_rest`,
`d_rest_build`, `d_builder`) re-exported, and where to find it
in the unified `d_rocket 2.0.0` (or `d_rocket_builder 2.0.0`
for codegen).

> If you already migrated to `d_rocket 1.x` and only need the
> 1.x → 2.0.0 changes (engine split, lint codes, realtime
> annotation rename), see
> [11-migration-1-x-to-2-0.md](11-migration-1-x-to-2-0.md)
> instead.

## Step 1: Update `pubspec.yaml`

```yaml
# Before
dependencies:
  d_serializer: ^1.3.0
  d_rest: ^0.1.0
  http: ^1.2.2

dev_dependencies:
  d_serializer_builder: ^1.2.0
  d_rest_build: ^0.1.0
  d_builder: ^0.1.0
```

```yaml
# After
dependencies:
  d_rocket: ^2.0.0

dev_dependencies:
  d_rocket_builder: ^2.0.0
```

> The `http` dependency no longer needs to be listed
> separately: `d_rocket 2.0.0` declares it transitively.

## Step 2: Update Dart imports

```dart
// Before
import 'package:d_serializer/d_serializer.dart';
import 'package:d_serializer_builder/d_serializer_builder.dart';
import 'package:d_rest/d_rest.dart';
import 'package:d_rest_build/d_rest_build.dart';
import 'package:d_builder/d_builder.dart';
```

```dart
// After
import 'package:d_rocket/d_rocket.dart';               // runtime
// codegen import is rarely needed in user code; the
// `d_rocket_registry.g.dart` is imported as a `part of`
// or by file path.
```

## Step 3: Update `part` directives

If you have any user file that uses the record codegen, the
serializer codegen, the REST codegen, or the ORM codegen,
update the `part` directive to the new suffix:

| Old `part` directive | New `part` directive | Builder |
|---|---|---|
| `part 'foo.g.dart';` (serializer) | `part 'foo.d_rocket_serializer.g.dart';` | `d_rocket_builder:serializer` |
| `part 'foo.g.dart';` (rest client) | `part 'foo.d_rocket_rest_client.g.dart';` | `d_rocket_builder:rest_client` |
| `part 'foo.g.dart';` (record) | `part 'foo.g.dart';` *(no change)* | `d_rocket_builder:record` |
| *(no `part`)* | `part 'foo.d_rocket_orm.g.dart';` | `d_rocket_builder:rocket_table` |
| `part 'foo.g.dart';` (legacy `d_builder:rest_client`) | `part 'foo_client.g.dart';` *(different — this was a `*_client` suffix in `d_builder`)* | — |

> **Why the suffixes?** The non-default suffixes
> (`.d_rocket_*.g.dart`) fix the long-standing build_runner
> output-collision bug. A single source file can now freely
> mix `extends Record` + `@Serializable` + `@RestClient` +
> `@Table` without any build error.

## Step 4: Replace `initializeD*()` with the central `initializeD()`

The five legacy packages each generated their own
`initializeD*()` function:

| Old call | New call |
|---|---|
| `initializeDSerializer()` (from `d_serializer_registry.g.dart`) | `initializeD()` (from `d_rocket_registry.g.dart`) |
| `initializeDClient()` (from `d_rest_build` codegen) | `initializeD()` |
| `initializeDBuilder()` (from `d_builder:registry`) | `initializeD()` |

The new `initializeD()` is the **single entry point** that
wires up every `extends Record` + `@Serializable` +
`@RestClient` + `@Table` class in your project. It is
generated by `d_rocket_builder:record_registry` as
`d_rocket_registry.g.dart` at the root of your `lib/`
directory.

```dart
// Before
import 'package:d_serializer/d_serializer_registry.g.dart';
import 'package:d_rest/d_rest_registry.g.dart';

void main() {
  initializeDSerializer();
  initializeDClient();
  runApp(const MyApp());
}
```

```dart
// After
import 'package:my_app/d_rocket_registry.g.dart';

void main() {
  initializeD();  // wires up EVERYTHING (records + serializers + REST + ORM)
  runApp(const MyApp());
}
```

## Step 5: Side-effect: `dRest.useDefaults()` is now idempotent

`d_rocket 2.0.0`'s `initializeD()` internally calls
`dRest.useDefaults()` to bootstrap the REST runtime. If your
app had its own `dRest.useDefaults(httpClient: ...)` call, you
can keep it — the new `useDefaults()` is idempotent and
respects any customization the user has made. Or you can
move all your customization into the central `initializeD()`
and skip the `useDefaults()` call.

## Step 6: Side-effect: `@Table` is a NEW feature (not a rename)

`@Table` is a new annotation introduced after the legacy
packages were frozen. It does **not** have a legacy
equivalent. If you have an existing class that you
manually-keep in sync with a SQL table, you can now annotate
it with `@Table()` and let the codegen generate the
`EntityMeta` + `register<X>EntityMeta()` for you. The
codegen runs as part of the same `dart run build_runner
build` that generates the rest of the code; no extra setup
is needed.

## Why this migration is worth doing

- **One version, one source of truth**: no more "is
  `d_serializer_builder 1.2.0` compatible with `d_rest_build
  0.1.0`?"
- **One central `initializeD()`** wires up every annotation
  kind in a single call.
- **Codegen output namespaces don't collide** (the
  `.d_rocket_*.g.dart` suffixes).
- **The `@Table` ORM is now available** alongside your
  existing code, and works on three engines
  (`d_rocket_engine_sqlite`, `d_rocket_engine_postgres`,
  `d_rocket_engine_web`).

## How long do the legacy packages stay in the monorepo?

The five legacy packages are frozen at their last public
version (`d_serializer 1.3.0`, `d_serializer_builder 1.2.0`,
`d_rest 0.1.0`, `d_rest_build 0.1.0`, `d_builder 0.1.0`).
They will remain in the monorepo **indefinitely** as
migration reference and emergency fallback. The
`discontinued:` flag in each `pubspec.yaml` and the
`DEPRECATED.md` banner tell pub.dev to mark them as
discontinued.

No removal date is scheduled.

## Release cadence

The `d_rocket` 2.x line ships a new release on the **first
Tuesday of every month** (or the next business day if that
Tuesday is a holiday in the maintainer's locale). Bug fixes
can ship at any time as patch releases of the latest minor
version. See
[RELEASE_CADENCE.md](RELEASE_CADENCE.md) for the full policy
and the next scheduled release date.
