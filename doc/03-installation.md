# 03 — Installation

`d_rocket` is published on [pub.dev](https://pub.dev/packages/d_rocket)
under two cooperating packages:

- **`d_rocket`** — the runtime (always required).
- **`d_rocket_builder`** — the codegen (always required, as a
  dev dependency).

This guide walks through the exact pubspec setup, the
build_runner integration, the platform-specific dependencies,
and the most common pitfalls.

## Supported platforms

`d_rocket` supports the five non-Web targets only:

| Platform | Status |
|---|---|
| Android, iOS, Linux, macOS, Windows | ✅ Supported (the ORM uses `package:sqlite3` + `dart:ffi`). |
| Web | ❌ **Not supported.** The `sqlite3` package wraps `dart:ffi`, which the Dart-to-JS compiler does not implement. The runtime and the LINQ provider work everywhere, but the ORM (Layer 4) does not. See the README's [Platform support](../README.md#platform-support) section for the full rationale and the Web roadmap. |

This is a hard limitation of `dart:ffi` itself, not a d_rocket
choice. It is unlikely to be lifted for the v1.x line. If your
project targets Web, you can still use Layers 1, 2, 3, 5, and 6
(serialization, REST, LINQ, sync, realtime); you just have to
bring your own storage (IndexedDB, OPFS, sql.js, etc.).

## 1. The pubspec

A minimal `pubspec.yaml` for a project that uses serialization,
REST, and the ORM:

```yaml
name: my_app
description: My first d_rocket app
publish_to: none

environment:
  sdk: ^3.5.0
  flutter: ">=3.10.0"

dependencies:
  d_rocket: ^2.0.0

dev_dependencies:
  d_rocket_builder: ^2.0.0
  build_runner: ^2.4.13
```

**Notes:**

- `d_rocket` is a **runtime** dependency — it ships in your
  app. The Flutter SDK constraint is `>=3.10.0`.
- `d_rocket_builder` is a **dev** dependency — it never
  ships with the app. It is the `build_runner` integration.
- `build_runner` is the standard Dart codegen runner. It
  orchestrates the codegen phases.

If you only need JSON serialization (no REST, no ORM), you can
still add `d_rocket` as a runtime dependency and skip
`d_rocket_builder`. The serializer registry is initialized
via the central `initializeD()` call which is generated by
`d_rocket_builder`. If you don't use any annotations, you can
import `package:d_rocket/d_rocket.dart` directly without
codegen.

## 2. Run the generator

After every change to an annotated class, run:

```bash
dart run build_runner build --delete-conflicting-outputs
```

The `--delete-conflicting-outputs` flag tells the runner to
delete any previously generated `*.g.dart` that no longer
matches the source. **Always** use this flag in development.

For continuous regeneration while editing, use:

```bash
dart run build_runner watch --delete-conflicting-outputs
```

The runner watches the source tree and re-emits the
generated files as you save.

## 3. Verify the registry

After the first build, your project should have a
`lib/d_rocket_registry.g.dart` file. Verify it has an
`initializeD()` function:

```dart
// lib/d_rocket_registry.g.dart
import 'package:my_app/todo.dart';
// ... other generated imports

void initializeD() {
  registerTodoSerializer();
  // ... other generated registrations
}
```

Call it from `main()`:

```dart
import 'package:my_app/d_rocket_registry.g.dart';

void main() {
  initializeD();
  // ... rest of your app
}
```

If `initializeD()` is missing, the codegen did not find any
annotated classes in the project. The most common reason is
that the annotation is on a class the codegen doesn't see
(e.g. a class in `lib/` that is not exported from
`lib/my_app.dart`).

## 4. The platform plugins

`d_rocket` uses `package:sqlite3` to talk to SQLite. The
plugin resolves the platform's SQLite library automatically
on every supported platform.

### Android, iOS, macOS, Windows, Linux

No extra setup. `package:sqlite3` links against the system
SQLite at runtime.

### Linux specifics

On some Linux distributions the system SQLite is older than
what `package:sqlite3` expects. If you see a runtime error
about a missing SQLite symbol, install a newer SQLite:

```bash
# Debian / Ubuntu
sudo apt install libsqlite3-dev

# Fedora
sudo dnf install sqlite-devel

# Arch
sudo pacman -S sqlite
```

### macOS specifics

No setup needed. macOS ships with SQLite.

### Windows specifics

No setup needed. `package:sqlite3` ships with a prebuilt
DLL.

### Web (JS) — NOT supported

`package:sqlite3` is a native library; it cannot run in a
browser. The framework is therefore **not supported on the
web** at this time. If you need a web build, you have two
options:

1. Use an in-memory `Db` for tests (and ship a
   different data layer for the web).
2. Build a custom `AsyncQueryProvider` that talks to
   IndexedDB and ship it as a `d_rocket_provider_*` package.

## 5. The lint rules

In 2.0.0 the lints moved to a separate
[`d_rocket_lints`](https://pub.dev/packages/d_rocket_lints)
package, implemented on top of
`package:analysis_server_plugin` (the modern
replacement for `custom_lint_builder`). Add it to
your dev_dependencies and enable the plugin in
`analysis_options.yaml`:

```yaml
dev_dependencies:
  d_rocket_lints: ^2.0.0
```

```yaml
# analysis_options.yaml
analyzer:
  plugins:
    - d_rocket_lints
```

The two diagnostic codes (the names the analyzer
emits) are:

- `d_rocket_n_plus_one` — flags LINQ queries that
  trigger N+1 round-trips. Promotes the
  `include_<TNav>()` codegen helper.
- `d_rocket_untranslated_closure_linq` — flags LINQ
  operators used on raw `Iterable<T>` with a closure
  (these evaluate in-memory only and can't be pushed
  to SQL). The fix is to wrap the closure in
  `Expr.lambda(...)` so the LINQ provider can
  translate it. `dart fix` applies the change
  automatically.

The 1.x class names `LinqClosureRule` and
`NPlusOneRule` (and the typedefs `LinqClosureLint` /
`NPlusOneLint`) are still importable from
`package:d_rocket_lints/d_rocket_lints.dart` for
1.x source compatibility, but the analyzer uses the
diagnostic codes above.

To run a one-shot rewrite of an entire file with the
closure-sugar fix, use the CLI:

```bash
dart run d_rocket:closure transform-file lib/services/order_query.dart
```

## 6. Common pitfalls

### "The named parameter 'onCreate' isn't defined"

You're on a `d_rocket` version older than 1.0. Upgrade:

```yaml
dependencies:
  d_rocket: ^2.0.0
```

### "Too many positional arguments" / "Undefined class 'Expr'"

The codegen has not been run, or has been run on stale source.
Re-run:

```bash
dart run build_runner build --delete-conflicting-outputs
```

If the error persists, the `Expr` import might be missing
in your source. Add:

```dart
import 'package:d_rocket/d_rocket.dart';
```

at the top of every file that uses LINQ.

### "The type 'Todo' is not a subtype of type 'Record'"

`@Table` requires the entity class to `extends Record`.
The `Record` class is the framework's typed-marker base
class for entities. Add it:

```dart
@Table('todos')
class Todo extends Record {
  @PrimaryKey(autoIncrement: true) final int id;
  // ...
}
```

The codegen also emits a `Record.toString()` that includes
each field by name and value, which is useful for logging.

### "Cannot resolve `sqlite3`"

You're missing the platform plugin. `package:sqlite3` resolves
on every supported platform without extra setup. If you see
this error, the most likely cause is a corrupted pub cache.
Try:

```bash
dart pub cache clean
dart pub get
```

### "Codegen emits changes on every build"

If the generated `*.g.dart` changes on every run with no
source changes, your build environment has a non-determinism
issue. The most common cause is a clock skew on a CI machine.
The runner uses timestamps in the file metadata; if the
system clock is off, files can be considered "newer" or
"older" than they are.

To work around it, commit the generated `*.g.dart` files and
only regenerate them in CI on a clean checkout:

```bash
git clean -fdx
dart pub get
dart run build_runner build --delete-conflicting-outputs
```

### "I have a circular import between codegen output and my source"

This is a known issue with `build_runner` when the generated
file imports from your source and your source imports from
the generated file. The fix is to put the generated file
in a `*.g.dart` and use a `part` directive or a separate
import. The `d_rocket_registry.g.dart` file is designed to
be imported separately:

```dart
// In your main():
import 'package:my_app/d_rocket_registry.g.dart';
```

without an import of the source classes. The codegen emits
a `register<X>` call per source class inside `initializeD()`,
so the source classes are referenced from the generated
file, not the other way around.

## 7. Upgrading from a pre-1.0 version

`d_rocket 0.x` was a pre-release. The 1.0 release is the
first stable version. Upgrading from a 0.x version requires:

1. Bump the version constraint:
   ```yaml
   dependencies:
     d_rocket: ^2.0.0
   ```
2. Re-run the codegen with `--delete-conflicting-outputs`.
3. Update test imports: `package:d_rocket/d_rocket.dart` (the
   path is unchanged, but some internal modules moved).
4. If you were using `d_rocket_provider_sqlite` directly,
   switch to depending on `d_rocket` instead — the engine
   is bundled in 1.0. (Note: `d_rocket_provider_sqlite` is
   also marked as deprecated; use `d_rocket` 1.2+ directly.)

The migration from `d_serializer` 1.3.0 and `d_rest` 0.1.0
to `d_rocket` 1.0 is documented in
[CHANGELOG.md](../CHANGELOG.md#migration-from-d_serializer--d_rest-0x).
The annotation API is unchanged; the only renames are
`Serializer.fromJson` → `Serializer.fromJson<T>` and
`Format` import path.
